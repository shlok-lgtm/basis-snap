## 1. What This Snap Does

The Basis Snap surfaces asset quality intelligence and wallet risk state at the point of financial decision — the MetaMask transaction confirmation window. It answers two questions no other wallet plugin asks:

1. **"Is this asset sound?"** — SII scores for any stablecoin involved in a transaction
2. **"Is this counterparty exposed?"** — Wallet risk profile for the recipient address (from the Wallet Risk Graph)

This is the first wallet plugin that scores **assets for quality**, not addresses for fraud. Every existing security Snap (Wallet Guard, Blockaid, etc.) asks "is this address/contract safe?" Basis asks "is the money itself sound?" and "what is this wallet's risk exposure?"

### 1.1 User Experience Summary

**At transaction time (onTransaction):**
- User initiates any transaction in MetaMask
- The "Basis Risk Intelligence" tab appears in the confirmation window
- If the transaction involves a scored stablecoin (to address is a known ERC-20, or data field contains a transfer call to a scored token), the Snap displays the SII score, grade, category breakdown, and a warning if below threshold
- The Snap also looks up the counterparty wallet (the `to` address) via the Wallet Risk Graph API and displays their risk profile: overall score, grade, concentration, coverage quality, dominant asset, and unscored exposure percentage
- If either the asset scores below the configurable threshold (default: 70) or the counterparty has low coverage quality, a warning banner is shown. If severity is critical (SII < 50 or counterparty risk score < 50), MetaMask shows a modal warning before the user can confirm

**On the home page (onHomePage):**
- User navigates to Snaps menu → selects "Basis Risk Intelligence"
- The home page displays the user's own wallet risk profile (fetched via the connected account address)
- Shows: overall risk score, grade, concentration HHI, coverage quality, unscored exposure, holdings breakdown with per-asset SII scores
- A "View All Scores" section shows the current SII leaderboard (top 10 stablecoins ranked by score)
- Risk score history sparkline (last 30 days) if available

**On installation (onInstall):**
- Welcome dialog explaining what the Snap does, with links to the Basis dashboard and methodology documentation

## 4. Entry Points — Detailed Behavior

### 4.1 `onTransaction` — Transaction Insights

**Trigger:** MetaMask calls this when the user views the Snap's tab in the transaction confirmation window.

**Input:** `{ transaction, chainId, transactionOrigin }`

**Logic flow:**

```
1. Extract `to` address from transaction
2. Check if `to` is a known stablecoin contract address
   → If YES: this is a direct stablecoin transfer
     → Fetch SII score for that stablecoin
   → If NO: decode `data` field for ERC-20 transfer/approve calls
     → Parse function selector (first 4 bytes)
     → 0xa9059cbb = transfer(address,uint256)
     → 0x095ea7b3 = approve(address,uint256)
     → If `to` matches a known stablecoin contract: fetch SII score
3. Look up counterparty wallet risk profile
   → For direct ETH/token transfers: counterparty = `to`
   → For ERC-20 transfers: counterparty = decoded recipient from calldata
   → Call GET /api/wallets/{counterparty}
   → If 404 (wallet not indexed): display "Counterparty not in Basis index"
4. Determine severity
   → "critical" if SII < 50 OR counterparty risk_score < 50
   → Warning banner if SII < threshold (default 70) OR coverage_quality = "low"
5. Return JSX UI content (and optional severity)
```

**Stablecoin detection — known contract registry:**

The Snap embeds a hardcoded map of known stablecoin contract addresses (Ethereum mainnet) to Basis coin IDs. This is the same list from `app/config.py` in Deploy-Guide:


**ERC-20 calldata decoding:**

```typescript
function decodeERC20Transfer(data: string): { recipient: string; } | null {
  if (!data || data === "0x" || data.length < 10) return null;
  const selector = data.slice(0, 10).toLowerCase();
  // transfer(address,uint256) or approve(address,uint256)
  if (selector === "0xa9059cbb" || selector === "0x095ea7b3") {
    const recipient = "0x" + data.slice(34, 74);
    return { recipient };
  }
  return null;
}
```

### 4.2 `onHomePage` — Wallet Risk Dashboard

**Trigger:** User selects "Basis Risk Intelligence" from the MetaMask Snaps menu.

**Logic flow:**

```
1. Get connected wallet address via ethereum.request({ method: "eth_accounts" })
   → Note: This returns accounts the Snap has access to
   → If no accounts: show "Connect your wallet" message
2. Fetch wallet risk profile: GET /api/wallets/{address}
3. Fetch all scores: GET /api/scores (for the leaderboard)
4. Fetch wallet history: GET /api/wallets/{address}/history?limit=30
5. Build and return interactive UI
```

**Home page sections:**

**Section A — Your Wallet Risk Profile**
- Risk score (0-100) with grade badge (A+ through F)
- Size tier label (whale / institutional / retail)
- Concentration HHI with grade
- Coverage quality indicator (full / high / partial / low)
- Unscored exposure percentage
- Dominant asset and its percentage of wallet
- Number of scored vs unscored holdings

**Section B — Your Holdings**
- Table: Symbol | Balance (USD) | SII Score | Grade | % of Wallet
- Unscored holdings shown with "—" for score, flagged visually
- Sorted by value descending

**Section C — SII Leaderboard**
- Top 10 stablecoins by SII score
- Columns: Rank | Name | Score | Grade | Daily Change

**Section D — Risk History**
- Last 30 days of daily risk scores (text-based, since Snaps don't support charts)
- Format: date | score | grade per row

### 4.4 `onCronjob` — Periodic Score Refresh

**Trigger:** Every 6 hours (cron: `0 */6 * * *`).

**Logic:**
```
1. Fetch GET /api/scores
2. Store in snap_manageState (encrypted)
3. Compare with previous cached scores
4. If any previously-above-threshold score dropped below threshold:
   → Send snap_notify (inApp) with expanded view:
     "USDT SII dropped from 87.5 to 64.2 (Grade: C)"
```

## 6. API Client Specification

### 6.1 Base Configuration

```typescript
const API_BASE = "https://basis-deploy-guide.replit.app";
const API_TIMEOUT = 10_000; // 10 seconds
const MAX_RETRIES = 2;
const RETRY_DELAY = 1_000; // 1 second
```

### 6.2 API Functions

```typescript
async function fetchScores(): Promise<ScoresResponse>
// GET /api/scores
// Returns: { stablecoins: [...], count: number, formula_version: string }

async function fetchScoreDetail(coin: string): Promise<ScoreDetail>
// GET /api/scores/{coin}
// Returns: full breakdown with categories, structural sub-scores

async function fetchWalletProfile(address: string): Promise<WalletProfile | null>
// GET /api/wallets/{address}
// Returns: { wallet: {...}, risk: {...}, holdings: [...] }
// Returns null on 404 (wallet not indexed)

async function fetchWalletHistory(address: string, limit?: number): Promise<WalletHistory>
// GET /api/wallets/{address}/history?limit=30
// Returns: { address: string, history: [...] }

async function fetchHealth(): Promise<HealthResponse>
// GET /api/health
// Returns: system status
```

### 6.3 Error Handling

```typescript
async function apiFetch<T>(path: string): Promise<T | null> {
  for (let attempt = 0; attempt <= MAX_RETRIES; attempt++) {
    try {
      const response = await fetch(`${API_BASE}${path}`, {
        method: "GET",
        headers: { "Accept": "application/json" },
      });
      if (response.status === 404) return null;
      if (!response.ok) throw new Error(`HTTP ${response.status}`);
      return await response.json() as T;
    } catch (error) {
      if (attempt < MAX_RETRIES) {
        await new Promise(r => setTimeout(r, RETRY_DELAY * (attempt + 1)));
        continue;
      }
      return null;
    }
  }
  return null;
}
```

**Degradation behavior when API is unreachable:**
1. Check local cache (snap_manageState) for last-known scores
2. If cache exists and is < 24 hours old: display cached data with timestamp and "⚠ Using cached data — API unreachable" banner
3. If cache is stale or missing: display "Basis API is currently unavailable. Scores could not be retrieved." — no severity flag, no blocking

## 7. Data Caching Strategy

**What gets cached (via `snap_manageState`):**

```typescript
interface CachedState {
  scores: {
    data: ScoresResponse;
    fetchedAt: number; // Unix timestamp
  } | null;
  walletProfiles: Record<string, {
    data: WalletProfile;
    fetchedAt: number;
  }>;
  settings: {
    warningThreshold: number; // default: 70
    criticalThreshold: number; // default: 50
  };
  version: string; // for migration
}
```

**Cache TTL:**
- Scores: 6 hours (refreshed by cronjob)
- Wallet profiles: 1 hour (per-address)
- Settings: permanent until user changes

**Cache size management:**
- Max 50 wallet profiles cached (LRU eviction)
- Total state should stay well under snap_manageState limits

## 11. Security Considerations

### 11.1 Data Privacy

- **No PII collected.** The Snap reads the user's public wallet address (already visible on-chain) and sends it to the Basis API. No names, emails, or off-chain data involved
- **API logging.** The Basis API may log request IPs and wallet addresses queried. This should be disclosed in the Snap's privacy description. Consider adding a note: "Queried addresses are logged for rate limiting and abuse prevention only"
- **Local state is encrypted.** `snap_manageState` encrypts cached data at rest within MetaMask

### 11.2 Rate Limiting

- **Current API has no rate limiting.** This is a risk if the Snap gets significant adoption
- **Recommendation:** Add API rate limiting before Snap launch (per-IP: 100 req/min, per-endpoint: 30 req/min)
- **Snap-side mitigation:** Cache aggressively. The 6-hour cronjob refresh means the /api/scores endpoint gets hit at most 4x/day per user. Wallet profiles are cached for 1 hour. Transaction insights may generate 1-2 API calls per transaction (score lookup + wallet lookup)

### 11.3 API Availability

- **Single point of failure.** The Snap depends entirely on the Replit-hosted API
- **Mitigation:** Cached fallback for scores (max 24-hour staleness). Wallet lookups degrade gracefully (show "not indexed" rather than error)
- **Future:** Consider a redundant API endpoint or CDN-cached scores endpoint
- **Never block transactions.** If the API is completely unreachable and cache is empty, the Snap shows an informational message but does NOT set severity to "critical" — it should never prevent a user from completing a transaction

### 11.4 Data Integrity

- **Scores are read-only.** The Snap cannot modify scores or wallet profiles
- **No authentication required.** The API is public. This means anyone can query it, which is intentional (permissionless distribution)
- **Score staleness.** Display the `computed_at` timestamp prominently so users know data freshness

### 11.5 Supply Chain Security

- **Minimal dependencies.** The Snap should use only `@metamask/snaps-sdk` and standard Snap tooling. No third-party libraries for API calls (use built-in `fetch`)
- **Lock dependencies.** Use exact versions in `package.json`, not ranges
- **Publish with `--provenance`** flag to npm for supply chain attestation (npm provenance)

## Appendix B: Competitive Context

**Why this Snap is differentiated:**

| Existing Snap | What it does | Basis difference |
|---|---|---|
| Wallet Guard | Phishing detection, malicious contract flagging | Scores addresses for fraud; Basis scores assets for quality |
| Blockaid | Transaction simulation, scam detection | Simulates outcomes; Basis evaluates underlying asset integrity |
| Tenderly | Transaction preview/simulation | Shows execution results; Basis shows asset fundamentals |

No existing Snap answers: "Is this stablecoin actually sound?" That is the asset quality gap Basis fills.
